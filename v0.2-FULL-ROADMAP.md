## Recon findings (these change the plan)

Before sequencing, the P1 code recon delivered four surprises that materially shift effort and ordering:

### P1 #2 — Truncation: no internal cause
- No `.substring()` / `.slice()` on message content anywhere in extension or backend.
- `extractTextFromElement → domToMarkdown` walks the full DOM tree; only filter is `content.length >= 10`.
- Fastify `BODY_LIMIT = 10 MB` (`src/main/http.ts:11`) — orders of magnitude above any realistic conversation.
- **Remaining hypothesis:** Claude/ChatGPT lazy-virtualize off-screen turns out of the DOM, so the scraper only sees what's been rendered into view. **Cannot confirm or fix without a real saved file that exhibits the bug.** → **BLOCKED ON USER EVIDENCE.**

### P1 #3 — "Memories tab" already exists
- `Sidebar.tsx:7` defines `type SidebarTab = 'memories' | 'files'`; default is `'memories'`. The Sidebar already filters memory rows vs vault-file rows.
- The actual perceived bug: every captured chat appears in **both** tabs because two pipelines ingest it:
  1. Extension `POST /api/memories` → row in `memories` table → shows in Memories tab.
  2. Vault watcher indexes the saved `.md` from disk → row in `vault_files` table → shows in Files tab.
- → **This is largely a duplicate of P0 #1 (dedup).** Once dedup is cross-pipeline aware, a saved chat appears once. Net new effort in v0.2: ~1 hr to add `type: saved-chat` frontmatter filter on the vault-file side. Not a separate P1.

### P1 #4 — Connections are not random; memory↔memory edges are empty
- `graph-builder.ts:109-113` builds memory↔memory edges purely from the `relationships` table. **Zero code path auto-populates this table.** `createRelationship` is exposed via store/IPC but only callable manually.
- Memory↔file edges (`'mention'` type, line 115-125) **already use a semantic-ish heuristic**: a memory is linked to a file if its content contains the file's stem (`graph-builder.ts:120`).
- → "Connections appear random" is wrong. What's actually missing: **memory↔memory auto-edges**. New algorithm needed.

### Issue #1 — Already shipped (commit `4bb9686`)
- Indexing progress now surfaces in StatusBar + Settings. No further work.

## P1 #4 — Memory↔memory auto-edge algorithm (spec)

Lifted to specification level because it's the only genuinely new design work in this roadmap.

**Goal:** When a new memory is created, automatically create `relationships` rows linking it to existing memories that are clearly related, with a *named reason* per edge.

**Three signals, cheap-to-expensive:**

1. **Shared tags** (free) — if memory A and memory B share ≥2 tags, link with `relationship_type = 'shared-tags'`.
2. **Keyword Jaccard** (cheap) — extract title + first-paragraph tokens (stopworded, lowercased); if Jaccard similarity ≥ 0.3, link with `'keyword-overlap'`.
3. **Embedding cosine** (Ollama-dependent) — if both memories are embedded and cosine ≥ 0.75, link with `'semantic'`. Skip if Ollama unavailable; degrades gracefully to signals 1+2 (consistent with existing fallback pattern).

**Per-edge metadata** (new column `relationships.confidence REAL` + reuse existing `relationship_type`):
- `shared-tags` → confidence = `min(1.0, sharedCount / 3)`
- `keyword-overlap` → confidence = jaccard score
- `semantic` → confidence = cosine score

**Cap:** at most 5 auto-edges per new memory (top-N by confidence). Prevents the graph from becoming a hairball as the vault grows.

**Idempotency:** `createRelationship` should upsert on `(memory_a_id, memory_b_id)` ordered tuple. Re-running auto-link on an existing memory doesn't duplicate.

**Trigger points:**
- After a successful `POST /api/memories` (new capture).
- After embedding lands for a previously-unembedded memory (so semantic signal eventually catches up).
- Manual "rebuild graph" button in Settings (debug + recovery).

**UI surface (graph-builder.ts change):**
- Edge color: muted gray for `shared-tags`, slate for `keyword-overlap`, blue for `semantic`.
- Hover tooltip: `"shared-tags: ai, prompt-engineering"` / `"semantic: 0.81"`.
- Filter toggles in graph view: "show edges from [tags] [keywords] [semantic] [mentions]".

**Schema migration:**
```sql
ALTER TABLE relationships ADD COLUMN confidence REAL DEFAULT 1.0;
CREATE INDEX IF NOT EXISTS idx_rel_pair ON relationships (memory_a_id, memory_b_id);
```

**Tests:**
- Unit: `src/main/auto-edges.test.ts` — each signal in isolation, cap behaviour, idempotency.
- Integration: insert 10 memories, run auto-edge pass, assert plausible link counts (≥ N, ≤ N×5).

**Effort: 8-12 hrs.** Risk: Medium — the algorithm is straightforward but the "what counts as a good edge" judgment is subjective; expect tuning during dogfooding.
