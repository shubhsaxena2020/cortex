# V04-THINKING

Written 2026-06-13, after shipping v0.3.0 and before deciding v0.4. The
ROADMAP says v0.4 is "Polish + Distribution" — code signing, themes,
accessibility, auto-update. This document argues v0.4 should be something
else entirely.

## What v0.3 actually revealed

v0.3 was a great release on paper. Six of eight kill-criteria items shipped,
plus two off-roadmap landmines: an MCP server, and a 44× graph perf jump
that takes the canvas from 2.9 FPS at 100k nodes to 133 FPS settled.

But when I use the app for an hour, something is still off.

The graph is impressive. It is also not what makes me come back. I open the
graph, I look at it, I find it visually satisfying, and I close the app. I
do not act on what it shows me. The same is true of the memory list and the
search box: they work, but I have to consciously decide to use them, and
that decision is the bottleneck.

The capture pipeline works in the opposite direction: it's passive. Memories
arrive on their own. I never decide to "save" anything — the Chrome
extension does it. That asymmetry is the entire reason Cortex got built:
the moat is *capture without effort*. But after capture, retrieval reverts
to effort. I have to remember Cortex exists. I have to remember to look.

**The product is half-passive on intake and fully active on retrieval.**
That is the gap. Cortex captures effortlessly and then waits to be asked.
A second brain that waits to be asked is a filing cabinet. The thing it
needs to become is the thing that finds *me*.

The MCP server is the first crack in this. With it, Claude can reach into
my second brain mid-conversation. I don't have to open Cortex; I just ask
Claude something, and Claude pulls the memory. That's exactly the right
inversion. But it's underexploited: the MCP server returns full content,
which blows the LLM's context budget on long memories; it has no notion of
"what's relevant right now"; and it lives inside Claude conversations,
which is exactly one of three contexts I work in. The other two are my
terminal and my IDE.

The 13 seed memories tell their own story. They're all about Cortex itself.
Read together, they reveal a project that has been building infrastructure
faster than it has been building habits. There is no memory about the work
the user is actually doing day to day — because the user has been working
*on* Cortex, not *with* it. v0.3 finally gave the app enough surface area
to be worth using. v0.4 has to make it sticky.

## What I'd build if I were the user

If I were a solo dev with Cortex installed and Ollama running, here are the
things I'd actually want, in priority order. I'm trying hard to be honest
about what I'd reach for, not what sounds smart in a spec.

1. **A terminal command.** I live in a terminal. If I can type
   `cortex search auth` and get the top 5 memories in two seconds, I will
   use Cortex twenty times a day. If I have to open an Electron app, I will
   use it twice a week. This is the highest-leverage single thing I could
   build. It also costs almost nothing because the MCP server already
   exposes everything; the CLI is the MCP server's twin with a different
   I/O surface, sharing the same pure core.

2. **Summaries.** Every memory should have a one-line and a paragraph
   summary, generated by local Ollama against `llama3.2:3b`. The MCP server
   should return summaries by default — `cortex_search` answers should be
   60 lines of summarized hits, not 600 lines of raw content that Claude
   has to re-summarize itself. This is the bandwidth fix. It also unlocks
   the digest below.

3. **A daily / weekly digest.** "Here's what you captured this week,
   grouped by topic." Auto-summarized via the same Ollama model.
   Pinnable, dismissible. Could be in the app, but the killer version is
   `cortex digest` from the terminal — a 10-line summary I read with my
   morning coffee, with links I can click open into the app for the ones
   worth looking at. This is what makes Cortex a habit instead of a tool.

4. **Pinned context.** Three to five memories I designate as "always
   relevant." They get prepended to every `cortex_search` result and shown
   at the top of the app sidebar. For me right now they would be: "Cortex
   architecture overview", "What I'm working on this week",
   "Open questions I haven't answered." This is the user-IS memory the
   13 seeds reveal as missing.

5. **Medium-zoom edge LOD.** v0.3 nailed far and close zoom but the medium
   band is still 78ms/frame with 40k edges visible. Auto-raise the strength
   threshold as visible-edge count rises. Five-line fix; uncaps the one
   remaining sad path.

## What I'd cut from the ROADMAP

- **Code signing ($300/yr + provisioning).** No real users yet. Sign when
  there are. Until then, GitHub-released unsigned installer + a one-line
  README warning beats nothing.
- **macOS DMG + Linux AppImage.** Same logic: zero users on those
  platforms today. Cross-compile when the ask is real.
- **Auto-update.** Requires signed builds to mean anything. Wait.
- **Export graph as PNG/SVG/PDF.** A novelty. Nobody will use this.
- **Dark/light theme toggle.** Dark is good. Single-theme is a feature, not
  a limitation, at this stage.
- **Keyboard navigation through graph.** Low marginal value; would have
  been important if the graph was the main surface, but the CLI + MCP
  shifts the main surface elsewhere.
- **Comprehensive accessibility pass.** Important when there are users.
  Right now it's a checkbox that doesn't ship anything.
- **`db.test.disabled.ts` → vitest-electron.** Infra; 399 unit tests +
  live-process verification scripts cover the layer. Punt to v1.0 with the
  benchmark suite.

I'm explicitly NOT cutting these because they're bad. They're future
table-stakes. They're just not what makes Cortex earn its keep against the
next month of my own use.

## v0.4 thesis: "Cortex finds you, not the other way around"

Three things, in this order:

### 1. `cortex` CLI (highest leverage)

A real command-line companion. Cross-platform, ships in the repo, lives on
PATH after `npm install -g .` or equivalent. Same DB, same FTS5, same
sqlite-vec, same Ollama. Same pure core (`mcp/core.mjs`) — the CLI is just
a different I/O layer over the existing query surface.

```
cortex search "auth flow"           # top 5 hits with snippets
cortex search "auth" --semantic     # force semantic mode
cortex recent --tag rust --limit 20
cortex digest                       # today's digest, summarized
cortex digest --weekly
cortex export <id>                  # markdown to stdout
cortex stats                        # what does my brain look like
cortex tags                         # vocabulary + counts
cortex pin <id>                     # add to pinned context
cortex unpin <id>
```

That's the v1. Pipeable. Greppable. Aliasable. Shell-completion-able later.
This is what turns Cortex from a thing I open to a thing I use.

### 2. Ollama summarization with `llama3.2:3b`

Two summaries per memory:
- One-liner (≤20 words): for list cells and CLI output.
- Paragraph (≤80 words): for detail panels and MCP responses.

Generated on demand the first time a summary is needed; cached in a new
`memory_summaries` table keyed by memory id + content hash so summaries
invalidate when the memory is re-captured. Skipped silently if Ollama or
the model is absent — same graceful-degradation pattern as embeddings.

The MCP server changes: `cortex_search` returns summaries by default;
clients can opt into full content with `mode: "full"`. This is the
bandwidth fix that makes the MCP server actually composable with
larger context windows.

### 3. Daily / weekly digest

A view in the app AND a `cortex digest` command. Groups the past 24h
(or 7d) of captures by tag, summarizes each cluster, surfaces the top
3 themes. The killer move is that the CLI version is two-second
read material — designed for the morning-coffee use case.

### Plus: the medium-zoom edge fix

Five-line tweak on top of v0.3's graph work. Raise the strength
filter from 0.2 to a sliding value when visible-edge count crosses
some threshold. Buys back the medium band without breaking far/close.

### Pinned memories + bonus tactical wins

Pin support: a `pinned` BOOLEAN column on `memories` (or a tiny
`pinned_memories` join table — leaning toward the column for
simplicity since the pin set is small). Top of sidebar; prepended to
search; surfaced in the CLI's recent.

If I have leftover budget after the four core items: a `cortex chat`
interactive REPL that opens a continuous semantic-search session.
Type a query, get hits, refine, repeat. This is what makes the CLI
feel alive — but I'll only build it if the core ships clean.

## Why not the slipped items first?

The ROADMAP slipped three items from v0.3 — summarization, multi-model
picker, vitest-electron. Summarization is in this thesis (item #2);
multi-model picker is a settings UI with no user pull until summarization
exists (it gets pulled along for free); vitest-electron is infra that
doesn't move the product forward. So one stays slipped, the other two
ride along.

## Out of scope (calling it explicitly)

- No marketing site, landing page, or screencast in v0.4. Build the
  thing first.
- No published Chrome Web Store listing. Load-unpacked extension is fine
  for the audience of one (the developer).
- No P2P sync. v0.5 is where Connect lives.

## How I'll measure if v0.4 worked

Three pass/fail criteria, honest:

1. After a week of v0.4, have I used the CLI from a shell at least 20
   times? If no, the CLI failed.
2. After two weeks, have I read at least 5 daily digests without being
   reminded? If no, the digest failed at being habit-forming.
3. Have any summaries been so bad that I disabled the feature? If yes,
   `llama3.2:3b` is the wrong model and I need to revisit.

I'm not measuring tests passing or build green; those are the floor, not
the ceiling.

## What this means for v0.5

v0.5 was scoped as "Connect — own machines, not other people's" (digest,
local web companion, P2P sync). With the digest now in v0.4, v0.5 has
room. v0.5 should be either:

- the local web companion (`cortex.local` on the LAN), which is the
  natural next step after the CLI, or
- something I won't see until I've used v0.4 for a week.

I'm going to write V05-THINKING.md after v0.4 ships, not before. That's
the whole point of this pattern.
