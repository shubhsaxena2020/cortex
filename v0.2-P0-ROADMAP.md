## Code-grounding deltas vs original scoping

These three corrections are why this roadmap differs from the initial scoping doc — they're not opinion, they're what's in the repo today:

1. **Graph is already canvas, not SVG.** `GraphCanvas.tsx` uses `<canvas>` + `getContext('2d')` with D3 force simulation drawing manually. The "canvas migration" the original scoping doc called the long pole **does not exist**. P0 #3 risk drops from High to Low-Medium; effort drops ~5 hrs.
2. **Dedup by URL has partial infrastructure.** `/api/vault/check-url` (`src/main/http.ts:279`) already reads vault frontmatter and answers "is this URL captured?" The extension's `background.js` likely calls it pre-capture. P0 #1 work is hardening + DB-backing this path, not greenfield.
3. **Schema doesn't store URL.** `memories` table (`database/schema.sql`) has `id, title, content, source, created_at, updated_at, importance, tags` — no `url`. Dedup needs a migration: `ALTER TABLE memories ADD COLUMN url TEXT`. Add to P0 #1 scope explicitly.

## Items

### P0 #1 — Conversation deduplication

**Problem.** Same conversation scraped twice → two memories, two graph nodes, two vault files. Within a week of real LLM use the graph becomes a landfill (Council's exact phrase).

**Concrete spec.**
- Schema migration: add `memories.url TEXT NULL` + index. Backfill from frontmatter on first launch after upgrade.
- Capture API: accept `first_turn_ts` and `last_turn_ts` (extension extracts from DOM; falls back to capture-time if missing).
- Dedup key: `sha256(source + '|' + canonical_url + '|' + bucket_5min(first_turn_ts))`. URL canonicalisation strips query-string trackers (`?utm_*`, `?fbclid`, anchor fragments).
- On POST `/api/capture`:
  - Compute dedup key.
  - If existing memory matches: **replace** content (newer wins), update `updated_at`, keep original `id` so graph nodes stay stable.
  - If no match: insert as today.
- Vault: rewrite the existing `.md` file in place. Don't accumulate `*-1.md`, `*-2.md`.

**Acceptance.**
- [ ] Same Claude chat captured twice → 1 memory row, 1 graph node, 1 vault file, latest content wins.
- [ ] Different chats with same title don't dedupe (URL distinguishes them).
- [ ] Migration runs on existing 0.1.0-beta vaults without data loss.
- [ ] Unit: `src/main/dedup.test.ts` — key generation, URL canonicalisation, 5-min bucketing edge cases.
- [ ] Integration: `scripts/integration-tests.mjs` captures same URL twice, asserts row count.

**Effort: 8-11 hrs.** Migration + backfill is the hidden cost the original estimate missed.

**Commit:** `feat(v0.2): conversation deduplication by url+timestamp`

### P0 #2 — Smart capture filtering

**Problem.** Extension grabs system prompts, tool-call JSON, empty error responses. Noise pollutes search.

**Concrete spec.**
- Filter runs in `extension/content.js` *before* POST — never index junk in the first place.
- Provider-specific rules (Claude / ChatGPT / Gemini DOMs are different; current content.js already has per-provider branches at ~lines 118-264):
  - Skip DOM nodes matching system-prompt selectors per provider.
  - Skip blocks recognised as tool-call JSON (regex `^\s*\{.*"tool_use".*\}\s*$` or provider-specific class).
  - Skip empty / single-character responses.
- Keep raw capture in `_raw_content` field (vault frontmatter only, not indexed) for one release so users can see what was dropped; remove in v0.3.
- Per-provider fixture corpus (`tests/fixtures/capture-{claude,chatgpt,gemini}.html`) — sanitised real HTML snippets.

**Acceptance.**
- [ ] "You are Claude" / "You are ChatGPT" / similar system blocks filtered out.
- [ ] Tool-call JSON blocks filtered out.
- [ ] Empty / error-only captures rejected client-side with a status pill in the extension popup.
- [ ] Unit: `src/renderer/src/utils/capture-filter.test.ts` against fixtures per provider.
- [ ] Integration: one real fixture per provider, captured-vs-expected diff.

**Effort: 6-9 hrs.** Lower than original because rules engine is small; fixture curation is the variable cost.

**Commit:** `feat(v0.2): smart capture filtering (system prompts, tool calls, empties)`

### P0 #3 — Graph LOD + viewport culling

**Problem.** Graph slows past ~8000 nodes. Pan/zoom drops frames; first paint takes seconds.

**Reality check.** GraphCanvas.tsx is already canvas-based with D3 force sim drawn manually. No SVG-to-canvas migration needed. The actual work is LOD + culling layered on the existing render loop.

**Concrete spec.**
- Build a d3-quadtree of node positions per simulation tick. Reuse, don't rebuild every frame — only on `tick` event and `nodes` change.
- Viewport culling: `quadtree.visit()` over current canvas-space rect (with ~10% margin) → only nodes inside are drawn. Edges only drawn if both endpoints visible.
- LOD by `d3.zoomTransform.k`:
  - `k < 0.3`: render top-N by `connections` count (cluster-ish; ~10% of nodes). Skip labels.
  - `0.3 ≤ k < 0.7`: render visible nodes, no labels.
  - `k ≥ 0.7`: render visible nodes + labels.
- No force-sim work in a web worker — defer to v0.3 *unless* culling alone misses target. (Council kill criterion.)
- Instrument: `requestAnimationFrame` timestamps → rolling p95 frame time exposed in dev mode.

**Acceptance.**
- [ ] 8000-node test vault: pan/zoom at p95 < 16ms/frame (60fps) at zoom 0.5.
- [ ] First paint < 1s for 8000-node vault.
- [ ] Visual regression: 3 screenshots (k=0.2 / 0.5 / 1.0) checked into `tests/fixtures/graph-*.png`.
- [ ] Perf: `tests/perf/graph-fps.test.ts` runs 1000 simulated pan events, asserts p95.

**Effort: 6-9 hrs** (down from 11-15 — canvas migration removed from scope). **Risk: Medium** — fixture vault generation + reliable perf measurement is fiddlier than the LOD logic itself.

**Commit:** `perf(v0.2): graph LOD + viewport culling for 8000+ nodes`

### P0 #4 — Search latency <200ms p95

**Problem.** Semantic search round-trip slows linearly with vault size. Council target: p95 < 200ms on 10k memories.

**Concrete spec.**
- Step 1: **profile first, optimise second.** Instrument `embeddings.ts` + `db.ts` vector path with `performance.now()` deltas around: query-embedding (Ollama call), `vec_*` SQL, rerank/sort, transformer mapping. Log p50/p95 per stage on every search in dev mode.
- Likely bottlenecks (ordered by prior probability):
  1. Ollama embedding generation on cold model — fix: explicit warmup ping on app launch (`/api/embeddings` no-op).
  2. `vec_search` over an unindexed table — fix: confirm sqlite-vec index, check `EXPLAIN QUERY PLAN`.
  3. Embedding result count too high → expensive rerank — fix: `LIMIT 50` at the vec layer; rerank only top-50.
- Two timeout layers:
  - Ollama call timeout 150ms → falls back to FTS-only.
  - Whole search timeout 500ms → returns whatever's ready + degraded flag in response.
- Add seed script: `scripts/seed-10k-vault.mjs` generates a 10k-memory test vault for benchmarking.

**Acceptance.**
- [ ] `scripts/search-perf-test.mjs` runs 100 queries on the 10k seed vault, prints p50/p95/p99.
- [ ] Keyword (FTS5) p95 < 50ms on 10k vault.
- [ ] Semantic p95 < 200ms on 10k vault (Ollama warm).
- [ ] CI fails if regression > 25%.

**Effort: 7-10 hrs.** Profiling dictates how much optimisation work is real — could be 4 hrs if cold-Ollama is the whole answer.

**Commit:** `perf(v0.2): search latency <200ms p95 on 10k vault`

### P0 #5 — In-app feedback + opt-in local telemetry

**Problem.** Council: "0 users = 0 interviewees. Shipping is the research instrument." Need ground-truth signal on whether installs work, pairing works, capture works, users return.

**Hard constraint.** Never leaves the machine. Ever. This is the load-bearing brand promise.

**Concrete spec.**
- New table `usage_events (id INTEGER PK, ts INTEGER, event TEXT, payload TEXT)`. SQLite, same DB.
- Event vocabulary (closed set — review every addition):
  - `app_launch`, `extension_paired`, `vault_initialised`, `first_capture`, `capture_filtered`, `graph_opened`, `search_executed`, `error_recorded`.
  - Payload is JSON of counts/enums only. **No content, no paths, no URLs, no titles, no tag values, no error stacks containing user paths.**
- Default = OFF. First-launch prompt offers opt-in with a "show me what gets recorded" inline example.
- Settings → "Usage Logs" section: live table of last 100 events, "Export to JSON" button, "Delete All" button.
- Feedback button → opens `https://github.com/shubhsaxena2020/cortex/issues/new?template=feedback.md` with URL-pre-filled system info (OS, app version, Ollama status). Recent events appended client-side only if user clicks "Include logs."
- New template: `.github/ISSUE_TEMPLATE/feedback.md`.
- **Privacy audit test:** `src/main/usage-logs.privacy.test.ts` runs every test-suite event through regex scanners for paths (`C:\\`, `/Users/`), emails, UUIDs, URLs. Fails if anything matches.

**Acceptance.**
- [ ] Telemetry default-OFF on fresh install.
- [ ] Toggling on at T0 records events from T0 forward only (no retroactive).
- [ ] Settings page renders log table + delete button.
- [ ] Privacy audit test passes.
- [ ] Feedback button opens GitHub with pre-filled env info; no logs included unless user explicitly opts in per-issue.

**Effort: 9-12 hrs.** Lower than original because no server side — the entire backend is "INSERT INTO usage_events."

**Commit:** `feat(v0.2): in-app feedback + opt-in local-only telemetry`

## Rollout sequence

Sequenced for early de-risking and quick wins, not strict dependency:

| Week | Focus | Why this slot |
|---|---|---|
| 1-2 | P0 #1 dedup | Schema migration is the riskiest single change; do it first while the codebase is fresh from launch. |
| 3 | P0 #2 filtering | Composable with dedup — both are capture-pipeline cleanliness. |
| 4-5 | P0 #4 search latency | Profiling is fast; optimisation is the variable cost. Do before LOD so search instrumentation is in place. |
| 6-7 | P0 #3 graph LOD | Needs the 10k seed vault from P0 #4. Reuse the fixture. |
| 8 | P0 #5 feedback + telemetry | Light work; saves for last so events get instrumented in their final shape. |
| 9 | Integration, regression sweep, manual smoke on clean Win11 VM | Council #1's deferred blocker. |
| 10 | v0.2.0 release (not pre-release if all 5 land) | If only 4 of 5: ship as v0.2.0 with the missing item called out, push the 5th to v0.2.1. |

## Kill criteria

- **Any single item blocked > 3 days** → escalate via `/council-review` skill before resuming.
- **P0 #3 culling alone misses 16ms frame target** → web-worker force sim defers to v0.3; ship LOD-only and label the regression.
- **Combined effort > 65 hrs at week 6** → drop P0 #5, ship v0.2.0 with items 1-4, P0 #5 becomes v0.2.1.
- **No installs / no GitHub stars / no issues by week 4 of v0.2 work** → stop, return to Council for positioning rewrite (not feature rewrite). Outside-view check from Council #1.
- **First clean-VM smoke test fails on v0.1.0-beta installer** → freeze v0.2 work, fix v0.1.0-beta first as v0.1.1.

## Success metrics (T+30 days after v0.2.0)

- ≥ 2 unsolicited GitHub stars *or* 1 issue *or* 1 fork — any of those = signal the URL was worth occupying.
- Zero crash reports via in-app feedback channel.
- p95 search latency < 200ms on at least one real (non-seed) user vault that gets reported back.
- Graph stays responsive on the largest known user vault.

If none of those hit by T+30: the issue is positioning / discovery (README, screenshot, narrative), not the product. Fix that before more features.
